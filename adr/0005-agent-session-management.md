# ADR 0005: Project-Local Agent Session Management

**Date**: 2026-04-03

**Status**: Accepted

## Context

`cl9 agent spawn` currently launches Claude Code from within a project, but `cl9` does not yet own or track the Claude session identity. This creates several problems:

1. `--continue` is implicit and brittle. It fails when no conversation exists.
2. `cl9` cannot list, resume, or fork sessions it previously created.
3. `cl9` cannot distinguish a long-lived session from a one-off process launch.
4. Process state is not durable. If the wrapping `cl9` process dies unexpectedly, stale state is unavoidable unless we reconcile it later.

We want `cl9` to become the authority for agent session identity within a project while still using Claude Code as the runtime.

## Decision

Agent sessions and agent processes are **project-local state**.

They are stored in a SQLite database under:

```text
.cl9/state.db
```

This database is scoped to the current cl9 project only. Agent/session tables do **not** store project name or project path, because the project is already defined by the enclosing `.cl9/` directory.

The global XDG SQLite database remains responsible only for cross-project registry state such as `cl9 project register`.

## Terminology

**Agent Session**
- A durable Claude conversation identity.
- Identified by a UUID generated by `cl9`.
- Optionally has a human-friendly name for easier reference.
- Can outlive any individual Claude process.

**Agent Process**
- A single local Claude process launched by `cl9`.
- Attached to one session for the duration of that process.
- Owns the session while it is running.

## Session Identity

`cl9` generates the session ID before launching Claude:

1. Resolve the current project from `.cl9/`
2. Generate a UUID
3. Insert a row into `agent_sessions`
4. Insert a row into `agent_processes`
5. Launch Claude with `--session-id <uuid>`

`cl9` never scrapes Claude output to discover a session ID after the fact.

## Database Schema

### `agent_sessions`

```sql
CREATE TABLE IF NOT EXISTS agent_sessions (
    session_id TEXT PRIMARY KEY,
    name TEXT,                                   -- Optional human-friendly name
    tool TEXT NOT NULL,                          -- e.g. "claude-code"
    status TEXT NOT NULL,                        -- active, idle, resume_failed
    created_at TEXT NOT NULL,
    last_used_at TEXT NOT NULL,
    source_cwd TEXT NOT NULL,
    forked_from_session_id TEXT,
    metadata_json TEXT NOT NULL DEFAULT '{}'
);

CREATE INDEX IF NOT EXISTS idx_agent_sessions_name
ON agent_sessions(name) WHERE name IS NOT NULL;
```

### `agent_processes`

```sql
CREATE TABLE IF NOT EXISTS agent_processes (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT NOT NULL,
    pid INTEGER,
    status TEXT NOT NULL,                        -- starting, running, exited, failed_to_start, stale
    cwd TEXT NOT NULL,
    command_json TEXT NOT NULL,
    started_at TEXT NOT NULL,
    last_heartbeat_at TEXT NOT NULL,
    ended_at TEXT,
    exit_code INTEGER,
    FOREIGN KEY(session_id) REFERENCES agent_sessions(session_id)
);
```

### Invariants

At most one running process may own a session at a time:

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_agent_processes_one_running
ON agent_processes(session_id)
WHERE status IN ('starting', 'running');
```

Session ownership is therefore derived from `agent_processes`, not stored separately on the session row.

## Status Transitions

### Session Status

```
                    ┌──────────────┐
     spawn ────────►│    active    │◄──── continue (success)
                    └──────┬───────┘
                           │ process exits / stale reconciliation
                           ▼
                    ┌──────────────┐
                    │     idle     │
                    └──────┬───────┘
                           │ continue fails (Claude rejects resume)
                           ▼
                    ┌──────────────┐
                    │ resume_failed│
                    └──────────────┘
```

### Process Status

```
     starting ──────► running ──────► exited
         │                │
         ▼                ▼
  failed_to_start       stale
  (Claude didn't start) (heartbeat expired + PID gone)
```

## CLI Model

### `cl9 session list`

List known sessions for the current project.

This command is project-local and only works inside a cl9 project. Default output includes:

- Session ID (short prefix)
- Name (if set)
- Status
- Created time
- Last used time
- Whether a running process currently owns the session
- Fork parent (if any)

### `cl9 session prune`

Remove old idle sessions.

```bash
cl9 session prune                    # Default: idle sessions older than 30 days
cl9 session prune --older-than 7d    # Custom threshold
cl9 session prune --all              # All idle sessions
```

### `cl9 session delete <target>`

Explicitly delete a session by ID or name.

Refuses to delete sessions with a running process. Use `--force` to override (marks process as stale first).

### `cl9 agent spawn`

Create a new session and launch a new process bound to it.

```bash
cl9 agent spawn                      # New session, auto-generated name
cl9 agent spawn --name "auth-refactor"  # New session with name
```

Behavior:

1. Generate new UUID
2. Insert `agent_sessions` row (with optional name)
3. Insert `agent_processes` row with status `starting`
4. Launch Claude with:

```bash
claude --session-id <new-id> --name <name> ...
```

5. Update process row to `running` once child PID is confirmed
6. Start heartbeat loop (~5 minute interval)
7. On child exit, record `ended_at`, `exit_code`, update session to `idle`

`spawn` always means a new session. It does not imply `--continue`.

### `cl9 agent continue [<target>]`

Resume an existing session.

```bash
cl9 agent continue                   # Continue most recent session (latest)
cl9 agent continue latest            # Explicit latest
cl9 agent continue auth-refactor     # By name
cl9 agent continue abc123            # By ID prefix
```

Behavior:

1. If no target specified, default to `latest` (most recent by `last_used_at`)
2. Resolve target: match by name first, then by ID prefix
3. Refuse if session is currently owned by a running process:
   ```
   Error: Session 'auth-refactor' is currently owned by process 12345.
   Use 'cl9 agent fork auth-refactor' to branch from it instead.
   ```
4. Insert a new process row
5. Launch Claude with:

```bash
claude --resume <session-id> ...
```

6. Update timestamps/status on success or failure

### `cl9 agent fork <target>`

Fork an existing session into a new session.

```bash
cl9 agent fork auth-refactor                  # Fork by name
cl9 agent fork abc123                         # Fork by ID prefix
cl9 agent fork auth-refactor --name "auth-v2" # Fork with new name
```

Behavior:

1. Resolve target session (by name or ID prefix)
2. Generate a new UUID
3. Insert a new `agent_sessions` row with `forked_from_session_id=<old-id>`
4. Insert a new process row for the new session
5. Launch Claude with:

```bash
claude --resume <old-id> --fork-session --session-id <new-id> --name <new-name> ...
```

This creates a new tracked session lineage while preserving the source session unchanged. Forking is allowed even if the source session has a running process.

## Process Management

Agent execution uses `subprocess.Popen`, not `os.execvpe`.

Reason:
- `cl9` must remain alive to record child PID, maintain heartbeats, and record exit status.
- `exec` replaces the `cl9` process and makes durable process bookkeeping impossible.

### Terminal Attachment

The child process inherits the current terminal directly:

```python
proc = subprocess.Popen(
    cmd,
    stdin=None,    # Inherit from parent
    stdout=None,   # Inherit from parent
    stderr=None,   # Inherit from parent
    env=env,
    cwd=cwd,
)
```

This gives Claude direct terminal access for interactive use while `cl9` can still `wait()` on the child.

### Signal Forwarding

`cl9` forwards relevant signals to the child process:

```python
import signal

def forward_signal(signum, frame):
    if proc and proc.poll() is None:
        proc.send_signal(signum)

signal.signal(signal.SIGINT, forward_signal)
signal.signal(signal.SIGTERM, forward_signal)
```

This ensures Ctrl+C and termination signals reach Claude properly.

### Heartbeat Loop

While the child process is running, `cl9` updates `last_heartbeat_at` approximately every 5 minutes:

```python
HEARTBEAT_INTERVAL = 300  # seconds

while proc.poll() is None:
    # Wait for child or timeout
    try:
        proc.wait(timeout=HEARTBEAT_INTERVAL)
    except subprocess.TimeoutExpired:
        update_heartbeat(process_id)
```

The heartbeat provides distributed liveness evidence that survives PID ambiguity.

## Stale State Reconciliation

`cl9` is not guaranteed to observe every termination event. The wrapper itself may be killed, the machine may crash, etc.

Because of this, `agent_processes` is treated as durable but potentially stale state.

### Reconciliation Rules

On every command that reads or mutates session/process state (`session list`, `session prune`, `agent spawn`, `agent continue`, `agent fork`), `cl9` first reconciles process rows whose status is `starting` or `running`.

A process row is considered stale when **both** conditions are true:

1. `last_heartbeat_at` is older than 2× the heartbeat interval (10 minutes)
2. The recorded PID no longer exists (or cannot be verified)

This two-factor check reduces false positives from:
- PID reuse (new process with same PID)
- Temporary inability to check PID

When a process is marked stale:
- Set status to `stale`
- Set `ended_at` to current time (if empty)
- Leave `exit_code` as NULL (unknown)

When a session's only `starting`/`running` process becomes stale:
- Set session status to `idle`

### PID Liveness Check

```python
import os

def pid_exists(pid: int) -> bool:
    """Check if a process with the given PID exists."""
    try:
        os.kill(pid, 0)  # Signal 0 = existence check only
        return True
    except ProcessLookupError:
        return False
    except PermissionError:
        return True  # Process exists but we can't signal it
```

## Target Resolution

Session targets (for `continue`, `fork`, `delete`) resolve in this order:

1. **Keyword**: `latest` → most recent session by `last_used_at`
2. **Name**: Exact match on `agent_sessions.name`
3. **ID prefix**: Prefix match on `agent_sessions.session_id`

Ambiguous prefix matches produce an error listing the candidates.

## Scope Boundaries

### In Scope

- Project-local SQLite state for sessions and processes
- Explicit session IDs generated by `cl9`
- Session naming
- Session listing, pruning, deletion
- Spawn / continue / fork semantics
- Heartbeat-based liveness tracking
- Stale-process reconciliation

### Out of Scope

- Global list of sessions across all projects
- Transcript indexing or message-level storage
- Recovering sessions that were started outside `cl9`
- Cross-tool normalization beyond the `tool` field
- Session archival (vs deletion)

## Migration Strategy

Existing projects do not need data migration.

The project-local database is created lazily on first use of session management commands. Existing files such as `.cl9/config.json`, `.cl9/env/state.json`, and `.cl9/claude/CLAUDE.md` remain unchanged.

Moving more project-local state into SQLite later is compatible with this design, which is why the database is named `.cl9/state.db` rather than `.cl9/agent.db`.

## Consequences

### Positive

- `cl9` becomes authoritative for session identity
- `spawn` no longer depends on brittle implicit continue behavior
- Sessions can be listed, resumed, and forked deterministically
- Human-friendly session names improve UX
- Heartbeat mechanism provides robust stale detection
- Process rows provide operational visibility and cleanup hooks
- The design stays project-local and aligned with cl9's isolation model

### Negative

- Agent launching becomes more complex
- `cl9` must maintain heartbeat loop and reconciliation logic
- Background thread or async coordination needed for heartbeats
- Slightly higher resource usage (cl9 stays resident)

### Trade-offs Accepted

- Durability over simplicity: SQLite replaces implicit runtime-only state
- Project-local truth over global observability
- Heartbeat-based tracking over pure PID checks
- `cl9` as long-running wrapper rather than exec-and-exit
